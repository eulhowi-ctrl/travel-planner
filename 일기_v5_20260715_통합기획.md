## 0️⃣ 이 문서의 목적

지금까지 이 프로젝트 폴더에는 서로 다른 3세대의 설계가 섞여 있었음:

| 그룹 | 정체 | 위치 |
|------|------|------|
| **그룹 1** | JWT+PostgreSQL 완전판 (44~63개 API, Docker 배포) | 루트 MD 10개 + `backend/*.md` 5개 |
| **그룹 2** | SQLite 간소화판 (355~360개 여행지, 지도, 필터) | `local_setup/` |
| **그룹 3** | TRAVOL — AI개인화+커뮤니티, LocalStorage 기반 | `일기_v2~v4` |

**결정**: 새 프로그램은 **그룹 3(TRAVOL, v4)을 기반**으로 만든다. 단, 그룹 1·2의 문서에만 있고 TRAVOL에는 없는 유용한 기능들을 골라서 이번 버전(v5)에 흡수한다. 이 문서는 그 흡수 결과를 정리한 "통합 설계도"이며, 실제 코드는 아직 작성되지 않았음.

### ✅ 검토 후 확정된 결정 (2026-07-15)

- **백엔드**: JWT+PostgreSQL 도입하지 않고 **LocalStorage 방식 유지** (배포 간단함·서버비용 없음·즉시 공유 가능이 기기 간 동기화보다 우선)
- **환율 변환**: 고정 환율표가 아닌 **실시간 환율 API** 연동 (예: exchangerate-api.com 무료 티어). API 실패/오프라인 시 마지막 캐시값으로 폴백
- **작업 순서**: 6️⃣의 문서 순서 그대로 진행 (community-script.js → 헤더 통일 → 데이터셋 확장 → 지도 연동 → 예산관리 → QR코드 → 교통정보 → 자동일정생성 → 맞춤추천)

## 1️⃣ 유지 기반: TRAVOL(v4) 그대로 가져가는 것

- **브랜드/컨셉**: TRAVOL, AI 개인화(70%) + 커뮤니티(30%)
- **스택**: 순수 HTML5 + CSS3 + Vanilla JS(ES6+) + LocalStorage (백엔드 없이 정적 호스팅)
- **완성된 Phase 0~4**: 홈/검색/일정, 필터 4종+정렬, 갤러리, 위시리스트, 날씨(데모모드), 호스트프로필, 리뷰강화, 포인트, 챗봇, 예약, 패키지
- **LocalStorage 키 계약** (`wishlist`, `travolPoints`, `reviewVotes`, `bookings`, `selectedDestinationId`)
- **데모모드 패턴** (`WEATHER_CONFIG.apiKey`가 비면 데모데이터 반환 — 키 없이도 전체 기능 시연 가능한 재사용 패턴)
- **네이버 여행카페 링크 통합** (v2에서 시작해 v3까지 유지된 것 — 11개 회원수 1만+ 카페, 클릭 시 새 탭)
- **미해결 이슈 승계**: `community-script.js` 없음(커뮤니티 페이지 빈 화면), 헤더 네비게이션 11개 페이지 불일치 → v5 최우선 처리

## 2️⃣ 그룹 1(FastAPI 완전판)에서 흡수하는 것

TRAVOL은 백엔드가 없으므로, 아래 기능들은 **클라이언트 사이드(JS+LocalStorage)로 재구현**하거나 **외부 무료 API를 클라이언트에서 직접 호출**하는 방식으로 가져온다. 백엔드 자체(JWT/PostgreSQL/Docker)는 가져오지 않음 — 필요해지면 그룹 1 문서를 그대로 스펙으로 재활용 가능.

| 원본 기능 (그룹1) | TRAVOL v5 적용 방식 |
|---|---|
| 예산 관리 (`budgets.py`) — 카테고리별 추적, 실제 vs 예상, 7개국 통화 환율 변환 | `budget.html` 신설. LocalStorage에 예산 항목 저장, 카테고리별(항공/숙박/식사/활동/교통/쇼핑/기타) 집계. 환율은 **실시간 API**(예: exchangerate-api.com 무료 티어) 연동, KRW↔USD/JPY/CNY/THB/EUR 변환. API 실패/오프라인 시 최근 캐시된 환율로 폴백 |
| QR 코드 생성 (`qr_generator.py`, QR Server 외부 API) | 예약 확인/WiFi/연락처 QR을 `booking.html`, `itinerary.html`에 추가 — 백엔드 없이 `api.qrserver.com` URL을 클라이언트에서 직접 조립 |
| 교통 정보 통합 (`transportation.py`) — 장거리(항공/기차/버스/페리) + 지역(버스/지하철/택시) | `itinerary.html`의 "장소별 링크버튼"에 교통 수단 정보 및 이용 팁 섹션 추가 (정적 데이터로 시작, 도시별 매핑) |
| 자동 일정 생성 로직 (`itinerary_generator.py`) — 활동 자동 분배, 식사시간 배치 | 위시리스트→일정 변환 시 JS로 일자별 활동/식사 슬롯 자동 배치하는 함수 작성 (동일 알고리즘을 클라이언트에서 재현) |
| 맞춤형 추천 (`destinations/recommend`) — 프로필(테마/아이동반/선호활동) 기반 필터링 | 검색 페이지에 "내 프로필 기반 추천" 섹션 추가 — LocalStorage에 저장된 사용자 프로필로 클라이언트 필터링 |
| 구조화된 테스트 체크리스트 방식 (레벨1~4, 63개 항목) | v5 QA 단계에서 동일한 체크리스트 포맷 채택 (API 대신 "페이지별 기능 체크리스트"로 변형) |
| 보안 문서화 관행 (SQL Injection 방지, 소유권 검증 등) | 클라이언트 전용이라 해당 위협은 없지만, XSS 방지(innerHTML 대신 textContent), LocalStorage 데이터 검증 등 상응 체크리스트로 대체 |
| JWT 인증 + Bcrypt | **채택 보류** — v5는 계속 로그인 없는 LocalStorage 방식 유지. 실사용자 계정이 꼭 필요해지는 시점(예: 여러 기기 동기화)에 그룹1 문서를 그대로 백엔드 스펙으로 재사용 |
| Docker Compose 배포 | **채택 보류** — 정적 호스팅(Netlify/Vercel/GitHub Pages)이면 충분. 백엔드가 실제로 필요해지면 그룹1의 `docker-compose.yml` 재사용 가능 |

## 5️⃣ 아키텍처 원칙 (v5)

- **여전히 백엔드 없음**: 모든 상태는 LocalStorage. 확장이 필요해지면 그룹1 API 문서(`DESTINATIONS_API_GUIDE.md` 등)를 그대로 구현 스펙으로 사용
- **데모모드 패턴 일반화**: 날씨와 마찬가지로 환율도 실시간 API 연동 + 실패 시 마지막 캐시값으로 폴백. QR은 외부 무료 API(키 불필요)라 데모 자체가 불필요 — "API 실패 시 폴백" 원칙만 공통 적용
- **LocalStorage 키 계약 확장**: 기존 5종 + `budgetItems`(예산), `userProfile`(추천용 프로필), `generatedItinerary`(자동생성 일정) 추가 예정
