## 1️⃣ 세션 개요

일기_v7에서 목적지 선택 핸드오프 버그 4건(일정/예약/예산/챗봇)을 고치고 push까지 마친 상태였음. 이번 세션(v8)은 새 기능 논의로 시작:

> "여행지를 갈 때 실시간 이동수단에 대한 가격반영을 연동해야해. 비행기 왕복/편도를 네이버 항공에 연동한다던지, 숙소가격을 여기어때/네이버호텔/호텔스컴바인과 연동이 필요해"

이 요구사항을 실제로 구현 가능한 형태로 좁혀가는 논의를 거쳐, 최종적으로 네이버 딥링크 방식(Phase 1)만 구현하기로 확정하고 완료함.

## 2️⃣ 논의 과정 (구현 전 검토)

여러 방식을 순서대로 검토하고 전부 이유를 들어 배제/채택함:

| 방식 | 검토 결과 |
|---|---|
| 네이버/여기어때/호텔스컴바인 실시간 크롤링 | ❌ 이용약관상 자동 수집 금지, 게다가 정적 사이트라 크롤링 자체가 브라우저에서 불가능(백엔드 필요) |
| 스카이스캐너 API | ❌ 더 이상 개인 개발자용 셀프서비스가 아님 — 비즈니스 파트너십 심사 필요, RapidAPI의 비공식 래퍼는 약관 리스크 |
| 아마데우스(Amadeus for Developers) | 🟡 항공+숙소 둘 다 무료 티어로 즉시 가입 가능하지만, 국내 숙소(모텔/펜션/게스트하우스) 커버리지가 네이버 대비 매우 약하고 리뷰 생태계가 아예 없음(B2B API). 사용자가 "그럼 이거 쓰는 목적지 어디야?"라고 물었을 때 아직 아무 데도 연동 안 됐다는 걸 확인하고 **최종적으로 보류 결정** |
| 네이버 통합검색 딥링크 | ✅ **채택**. 실가격 데이터를 가져오는 건 아니지만, 목적지/도시명을 쿼리로 넘겨 네이버 자체 항공/호텔 비교 위젯으로 바로 연결. 백엔드 불필요, 지금 정적 사이트 구조 그대로 적용 가능 |

**기술적 검증 한계**: `flight.naver.com`의 정확한 딥링크 파라미터(도시코드, `fareType`, `selectedFlight` 등)를 웹 검색으로 찾긴 했으나, 이 환경에서 `flight.naver.com`과 `search.naver.com` 둘 다 브라우저 도구·WebFetch로 접근이 차단되어 있어 라이브 검증이 불가능했음. 틀릴 수 있는 세부 파라미터를 넣어 깨진 링크를 만드는 위험을 피하기 위해, 검증 없이도 항상 동작이 보장되는 **일반 검색 URL**(`search.naver.com/search.naver?query=`)만 사용하기로 결정.

## 3️⃣ 이번 세션에 구현한 것

### ✅ `script.js` — 공용 딥링크 헬퍼 추가
```js
function naverFlightSearchUrl(cityName) {
    return `https://search.naver.com/search.naver?query=${encodeURIComponent(cityName + ' 항공권')}`;
}
function naverHotelSearchUrl(cityName) {
    return `https://search.naver.com/search.naver?query=${encodeURIComponent(cityName + ' 호텔')}`;
}
```

### ✅ `search.html` — 목적지 상세 모달
기존 "🗺️ 네이버맵" 링크 옆에 "✈️ 항공권 가격 확인", "🏨 숙소 가격 확인" 버튼 추가.

### ✅ `itinerary.html` — 실시간 가격 확인 카드
날씨 위젯 바로 아래 새 섹션 추가. 선택된 목적지(`ITINERARY.city`)를 반영해서 도시명이 자동으로 바뀜.

### ✅ 라이브 사이트 검증
push 후 GitHub Actions 배포 성공(25초) 확인 → 실제 `eulhowi-ctrl.github.io/travel-planner`에서:
- 서울 선택 시 "서울 항공권" / "서울 호텔" 쿼리로 정확히 연결됨
- 부산 예약, 파리 패키지 예약(지난 세션 수정분) 등 다른 페이지들도 함께 재검증 — 전부 정상
