## 1️⃣ 세션 개요

사용자가 실제 서비스를 훑어보다가 "구현이 잘 안되거나 무의미한 기능들"을 발견. 특히 여행 후기 기능이 원래 의도와 다르다는 지적으로 시작:

> "후기 기능은 내가 의도한거랑 달라. 나는 후기를 직접 작성할 수 있는 기능을 넣으려고 한게 아님. 네이버 유명 카페들에서 여행지 후기 중 괜찮은것들 가져오려고한것임."

이걸 계기로 Opus 서브에이전트에게 전체 사이트 감사를 맡겼고, 예상보다 많은 문제(보안 이슈 포함)가 나와서 세션 전체가 "발견된 문제 하나씩 정리"로 진행됨. 감사는 이 세션에서 총 2회 실행됨 (1차: 전체 사이트, 2차: 딥링크 exact-match 버그 계열 재점검).

## 3️⃣ API 키 처리 방식 논의

사용자가 ".env + .gitignore로 숨기면 안되나"라고 질문 → 이 프로젝트는 빌드 단계가 없는 순수 정적 사이트라 `.env`가 로컬 개발용 메모 이상의 역할을 못 한다는 점을 설명. 대신:
- `weather-api.js`의 실키를 `__OPENWEATHER_API_KEY__` placeholder로 교체
- `.github/workflows/pages.yml`에 배포 시 `secrets.OPENWEATHER_API_KEY`로 치환하는 `sed` 스텝 추가
- `.env.example` 추가 (로컬 참고용)
- **사용자가 직접 해야 할 일**: OpenWeather 키 재발급(기존 키는 git 히스토리에 이미 노출되어 폐기 필요) + GitHub 저장소 Secret 등록. **이번 세션에서는 보류 결정** ("api는 미루자")

## 4️⃣ 후기 기능 재설계 — 크롤링 불가 확인 과정

사용자 의도(네이버 카페 우수 후기 3줄 요약)를 구현하려 했으나, 크롤링이 이 환경에서 근본적으로 불가능함을 재확인:
1. `WebFetch`로 `search.naver.com`/`flight.naver.com`/`cafe.naver.com` 전부 차단
2. Chrome 브라우저 확장 연결 후 재시도해도 `navigate` 자체가 "안전 정책상 차단" — naver.com은 도메인 카테고리 차원에서 전면 차단(재연결로 해결 안 되는 종류)
3. `WebSearch`로 외부 문서에서 파라미터 스킴을 찾으려 했으나 실패

→ 사용자가 "크롤링 포기, 딥링크 아이콘만 추가"로 결정. `naverReviewSearchUrl()` 헬퍼 추가, 일정 내 장소별 "카페 후기" 링크(기존엔 전부 고정된 가짜 URL `cafe.naver.com/firstlove`)도 장소별 검색 링크로 수정.

## 5️⃣ 항공/호텔/기차 딥링크 업그레이드

기존 Phase 1(v8)은 도시명 일반검색뿐이라 부족하다는 지적. naver.com이 전면 차단되어 있었지만, **사용자가 직접 Chrome에서 검색해서 실제 URL을 복사해주는 방식**으로 우회:
- `flight.naver.com` 국제선 왕복 실제 URL 확보 → `{출발지코드}:city-{도착지코드}:airport-{날짜}` 구조 확인
- 사용자가 직접 "부산은 비행기 타지 말고 KTX/SRT"라고 정정 → 국내 이동 수단 재설계
- 코레일(`korail.com/ticket/search/list`) 확인 결과 세션 기반 사이트라 파라미터화된 딥링크 자체가 불가능(검색해도 URL 안 바뀜) — 조회 페이지로만 연결하는 게 최선
- 네이버지도 자동차 길찾기는 목적지마다 네이버 내부 ID(예: `3zhjsT,2ALvNc`)가 필요해서 도시명만으로 직접 생성 불가 확인
- 사용자가 "시외버스터미널 기준"으로 방향 제시 → 처음엔 `map.naver.com/p/search/{도시} 시외버스터미널`로 구현했으나, **서울처럼 통일된 업체명이 없는 도시**(서울고속버스터미널/센트럴시티/남부터미널로 분산)에서 정확한 장소가 아닌 일반 도시 결과로 떨어지는 버그 발견 → `search.naver.com` 일반검색으로 전환(항공권/호텔/후기와 동일한 안정적 패턴)

**최종 정책**: 제주도(섬)는 항공권 딥링크 유지, 그 외 국내 목적지는 코레일 조회 + 시외버스터미널 검색, 해외는 기존 항공권 검색 유지.

## 6️⃣ 2차 Opus 감사 — exact-match 링크 버그 계열 재점검

시외버스터미널 버그를 계기로 사용자가 "opus야 전체 검토 다시해서, 서울처럼 딱 맞는 업체명이 없는 경우 같은 예외사항 찾아내"라고 요청. 감사 결과, `map.naver.com/p/search/{name}`가 **정확한 업체명이 아니면 조용히 일반 결과로 degrade**하는 문제가 코드 곳곳에 있었음:

- `itinerary.html` 자동 생성 일정(`buildGenericItinerary`)이 `dest.name`(패키지 마케팅명, 예: "도쿄 시티 투어")을 지도 링크에 사용 — 전체 ~300개 목적지 중 대부분(변형 destination)이 영향받는 최다 노출 버그였음. `dest.baseCity`로 수정
- `search.html` 목적지 상세 모달의 지도/항공권/호텔 링크도 동일 문제 → 이미 계산되어 있던 `cityKey`(baseCity)로 통일
- 위시리스트 기반 자동 일정(`item.name`)도 동일 문제 → 위시리스트에 저장된 id로 원본 destination을 다시 찾아 baseCity 사용
